一、选题背景与意义

高校师生日常既有图书借阅需求，也有教材、数码等闲置物品的二手流转需求。借阅若依赖纸质登记，易出现可借数量不清、审核慢、逾期难催；二手交易若仅靠群聊与线下摊位，则商品信息分散、成交过程难留痕。近五年专科、本科毕业设计中，「借阅管理系统」与「校园二手商城」均为高频选题；将二者置于同一 Web 门户、统一账号体系下的综合课题亦较常见，但须控制范围，避免功能堆砌。

本课题拟设计并实现一套基于 B/S 架构的校园图书借阅与二手交易系统，突出两条可答辩演示的主路径：图书借阅申请—审核—归还，以及二手商品浏览—加购—提交订单（演示环境不实现真实第三方支付），作为毕业设计工程实践训练。


二、国内外研究现状简述

图书馆业务系统与校园电商、二手交易平台在商业与开源领域均有成熟方案，但功能往往超出本科毕业设计可承受规模。国内毕设选题多采用 Spring Boot + Vue + MySQL，分别实现借阅或商城；将「单据审核流」与「购物车订单」同系统交付的案例近年增多，实现上需分清两套规则，不宜混为一谈。

本课题将研究范围限定在 Web 端的图书档案与借阅闭环、二手商品与订单办理、用户与公告管理，不扩展真实支付、物流轨迹、小程序原生交付及教务全模块等过重子系统。


三、研究目标

1. 分析校园借阅与二手交易两类业务，梳理读者、买家与管理员的使用需求。
2. 完成系统总体设计、模块划分与数据库设计。
3. 采用 Spring Boot 与 Vue 实现可运行系统，覆盖图书借阅主流程与二手购物车订单主流程。
4. 完成功能测试，整理源码、数据库脚本与毕业设计说明书，准备答辩。


四、研究内容与拟实现功能

（一）用户与权限

系统面向读者（兼二手买家）与管理人员。用户可注册登录、维护个人资料、检索图书并申请借阅、浏览二手商品并下单，查看本人借阅、订单与公告；管理侧负责图书与分类、借阅审核与归还、二手商品与订单办理，以及用户与公告管理。总管与业务办理岗位在详细设计中按职责划分。

（二）主要功能

1. 用户注册、登录（含验证码）与个人资料维护。
2. 图书分类浏览与条件检索，查看图书详情与可借数量。
3. 在线提交借阅申请，查询本人借阅记录；管理侧审核通过或驳回，支持归还登记；审核结果写入站内消息。
4. 二手商品分类浏览与上架，加入购物车并提交多明细订单。
5. 管理侧办理订单（确认或推进状态）；演示环境不实现微信支付、不实现支付宝对接。
6. 图书、商品与分类等基础数据管理。
7. 用户信息管理。
8. 校园公告的撰写与查阅。
9. 管理端业务概览（如待审借阅、近期订单等）。

（三）研究重点

围绕借阅单据状态变化（申请、审核、借出、归还）与购物车订单状态变化两条主路径，以及普通用户与管理人员的功能边界，形成可分别演示的完整业务过程。明确：借阅走审核流，二手走订单流，二者规则不同。


五、关键问题与解决思路

1. 需求范围控制：以借阅闭环与二手订单两条主路径及必要基础数据为主；不纳入真实支付、物流、小程序交付；不把第三条无关业务列为必交功能。
2. 数量一致性：借阅审核通过时扣减可借数量，归还回补；下单占用商品数量的规则在设计中约定清楚。
3. 权限与数据安全：区分门户与管理端，防止越权查看或办理他人单据、订单。
4. 前后端协作：明确接口与页面流程，保证联调与答辩演示可执行。


六、技术路线

1. 体系结构：B/S，前后端分离。
2. 后端：Java、Spring Boot、MySQL。
3. 前端：Vue、Element Plus 等。
4. 开发过程：需求分析 → 系统设计 → 编码实现 → 测试修改 → 撰写说明书。


七、进度安排（参考）

第 1–2 周：查阅资料，完成开题，细化需求与用例。
第 3–4 周：完成总体设计、数据库设计与界面草案。
第 5–8 周：实现登录注册、图书借阅、二手购物车订单、基础数据与公告等模块。
第 9–10 周：系统联调与测试，准备答辩演示。
第 11–12 周：完善毕业论文（设计说明书），准备答辩。


八、预期成果

1. 校园图书借阅与二手交易系统软件一套（含源码与数据库脚本）。
2. 毕业设计说明书一份。
3. 系统演示环境及主要操作说明（须能分别演示借阅审核与下单办理）。
