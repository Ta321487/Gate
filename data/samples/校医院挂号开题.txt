一、选题背景与意义

高校校医院、社区门诊普遍存在分时段号源管理需求。若挂号依赖窗口排队或纸质号条，容易出现号源信息不透明、重复占号、取消难以及时释放等问题，既增加候诊等待，也不利于医务管理侧掌握各科室接诊节奏。

建设面向校园场景的门诊挂号预约信息系统，将科室医生目录、分时号源、患者预约与取消、医院公告集中管理，有助于规范挂号流程、提高号源利用率。本课题拟设计并实现一套基于 B/S 架构的校医院门诊挂号预约系统，作为毕业设计工程实践课题，突出需求分析、系统设计与实现能力训练。


二、国内外研究现状简述

医院信息系统（HIS）、互联网医院等商业方案功能较全，但部署成本与业务范围往往超出本科毕业设计可承受规模。国内毕业设计选题里，基于 Java Web、Spring Boot 实现挂号、预约类系统的案例较为常见，技术路线成熟、可参考性强。

本课题结合校医院门诊场景，将研究范围限定在 Web 端的科室医生检索、分时号源预约与取消、患者资料与公告管理，不扩展电子病历、检验检查、药品处方、医保结算与真实第三方支付等过重子系统，保证在有限周期内完成分析、设计、编码、测试与说明书撰写。


三、研究目标

1. 分析校医院门诊挂号业务，梳理患者与医务管理侧的使用需求。
2. 完成系统总体设计、模块划分与数据库设计。
3. 采用 Spring Boot 与 Vue 实现可运行系统，覆盖科室医生目录、号源时段预约与取消、患者管理与公告等功能。
4. 完成功能测试，整理源码、数据库脚本与毕业设计说明书，准备答辩。


四、研究内容与拟实现功能

（一）用户与权限

系统面向患者与医务管理人员。患者可注册登录、维护个人资料、按科室检索医生并选择号源时段挂号，查看本人挂号记录与医院公告；管理侧负责医生与科室维护、号源时段生成与挂号记录查看，以及患者与公告管理。总管与挂号员职责在详细设计中按岗位划分。

（二）主要功能

1. 患者注册、登录与个人资料维护（含就诊相关基础信息）。
2. 按科室浏览与条件检索医生，查看职称/说明与挂号费等基本信息。
3. 选择医生可预约时段提交挂号，填写就诊人姓名；查询本人挂号记录。
4. 支持取消挂号并释放对应号源时段（按业务规则）。
5. 管理侧维护医生与科室等基础信息，并可为医生生成可预约号源时段。
6. 管理侧查看挂号记录，掌握各时段占用情况。
7. 患者用户信息管理。
8. 医院公告（挂号须知、临时停诊等）的发布与查阅。
9. 管理端业务概览（如医生数量、近期挂号情况等）。

（三）研究重点

围绕医生号源时段库存与预约单占用/释放，以及患者端与管理端功能边界，形成从选医生、选时段到挂号与取消的基本业务过程。


五、关键问题与解决思路

1. 需求范围控制：以医生目录、分时号源预约—取消主流程及必要基础数据为主；不纳入电子病历、处方开药、检验检查、排队叫号大屏、医保对接与真实支付。
2. 号源一致性：预约成功时占用时段库存，取消时回补，防止超卖；异常情况在设计中约定处理规则。
3. 权限与数据安全：区分患者端与管理端，防止越权查看或改动他人挂号。
4. 前后端协作：明确接口与页面流程，保证联调测试可执行。


六、技术路线

1. 体系结构：B/S，前后端分离。
2. 后端：Java、Spring Boot、MySQL。
3. 前端：Vue、Element Plus 等。
4. 开发过程：需求分析 → 系统设计 → 编码实现 → 测试修改 → 撰写说明书。

第 1–2 周：查阅资料，完成开题，细化需求与用例。

第 11–12 周：完善毕业论文（设计说明书），准备答辩。
