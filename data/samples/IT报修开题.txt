一、选题背景与意义

高校校园网与公共机房、教学终端等信息化设施使用频繁，断网、无法认证、设备异常等情况时有发生。师生报修若主要依赖电话、口头转告或分散的即时通讯群，容易出现描述不清、地点难定位、处理进度难以查询、事后缺少记录等问题，影响教学与办公秩序，也不利于信息中心掌握运维工作量。

建设面向校园场景的故障报修信息系统，将故障申报、处理跟进与通知公告集中管理，有助于提高响应效率与服务规范性。本课题拟设计并实现一套基于 B/S 架构的校园网故障报修管理系统，作为毕业设计工程实践课题，突出需求分析、系统设计与实现能力训练。


二、国内外研究现状简述

IT 服务管理领域中，事件单、服务台等理念较为成熟，商业运维平台功能丰富，但部署与学习成本较高，不完全适合本科毕业设计的规模控制。国内高校信息化建设普遍设有网络与信息中心，报修渠道逐步从线下转向线上；毕业设计中基于 Java Web、Spring Boot 实现报修或工单类系统的选题较为常见，技术路线可行。

本课题结合学校场景，将研究范围限定在 Web 端故障申报与处理闭环及必要的基础信息管理，保证在有限周期内完成分析、设计、编码、测试与说明书撰写。


三、研究目标

1. 分析校园网及公共终端故障报修业务，梳理师生用户与运维管理人员的使用需求。
2. 完成系统总体设计、模块划分与数据库设计。
3. 采用 Spring Boot 与 Vue 实现可运行系统，覆盖故障申报、处理与基础管理等功能。
4. 完成功能测试，整理源码、数据库脚本与毕业设计说明书，准备答辩。


四、研究内容与拟实现功能

（一）用户与权限

系统面向师生用户与运维管理人员。师生可注册登录、维护个人资料、提交故障报修并查看处理进度与相关通知；运维侧负责故障处理、区域与终端等基础信息维护，以及用户与公告管理。权限细节在详细设计中按岗位职责划分。

（二）主要功能

1. 用户注册、登录与个人资料维护。
2. 在线提交故障报修（选择区域、终端位置、故障类型，填写说明），查询本人报修记录。
3. 运维侧对待处理故障进行审核处理，并跟进至办结。
4. 校园区域、终端点位及故障类型等基础信息维护。
5. 用户信息管理。
6. 运维/信息中心公告的发布与查阅。
7. 管理端业务概览（如待办数量等）。

（三）研究重点

围绕故障信息的规范登记、处理状态变化，以及普通用户与管理人员的功能边界展开，形成从申报到办结的基本业务过程。


五、关键问题与解决思路

1. 需求范围控制：以故障报修主流程及必要基础数据为主，避免一次覆盖过多运维子系统。
2. 位置与分类表达：通过区域、终端点、故障类型等结构化信息辅助定位与分类，减少纯文字描述带来的歧义。
3. 权限与数据安全：区分师生端与管理端功能，防止越权访问。
4. 前后端协作：明确接口与页面流程，保证联调测试可执行。


六、技术路线

1. 体系结构：B/S，前后端分离。
2. 后端：Java、Spring Boot、MySQL。
3. 前端：Vue、Element Plus 等。
4. 开发过程：需求分析 → 系统设计 → 编码实现 → 测试修改 → 撰写说明书。


七、进度安排（参考）

第 1–2 周：查阅资料，完成开题，细化需求与用例。
第 3–4 周：完成总体设计、数据库设计与界面草案。
第 5–8 周：实现登录注册、故障报修、基础数据与公告等模块。
第 9–10 周：系统联调与测试，补充说明材料。
第 11–12 周：完善毕业论文（设计说明书），准备答辩。


八、预期成果

1. 校园网故障报修管理系统软件一套（含源码与数据库脚本）。
2. 毕业设计说明书一份。
3. 系统演示环境及主要操作说明。
