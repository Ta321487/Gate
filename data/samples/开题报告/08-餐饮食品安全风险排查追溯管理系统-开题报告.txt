本科毕业设计（论文）开题报告

题目：基于 Spring Boot 的餐饮食品安全风险排查与追溯管理系统的设计与实现

一、选题背景与意义

1.1 选题背景
食品安全关系人民群众身体健康与社会稳定。学校食堂、连锁餐饮、中央厨房等场景中，原料入库、加工过程、人员健康、消毒记录、异常处置与问题食品追溯是监管与自评的核心环节。现实管理中，很多单位仍用纸质表单记录晨检、消毒、留样与进货台账，存在补记代签、数据难查、风险难预警、问题难闭环等问题。

随着“互联网+明厨亮灶”、食品安全追溯与属地监管数字化推进，企业端也需要一套能支撑日常排查、人员健康监测、问题食品处置与物资台账的管理系统。

1.2 选题意义
（1）选题贴近民生与监管热点；
（2）业务闭环清楚：排查—发现—处置—追溯—统计；
（3）基于 Spring Boot 实现权限、业务与可视化完整系统；
（4）具备清晰的毕业设计工作量与演示路径。

二、国内外研究现状

发达国家在食品追溯方面强调全链条编码与责任到人；国内从中央到地方持续建设食品安全监管平台与企业端台账系统。中小餐饮或高校食堂仍需要轻量、好用、可私有化部署的管理工具。本课题定位企业/食堂内部风险管理系统，聚焦内部合规与效率提升。

三、研究目标与主要内容

3.1 研究目标
基于 Spring Boot 实现从业人员健康晨检、风险食品/批次管理、关联排查、处置闭环、检测抽检记录、消毒与防护物资管理、追溯查询与统计看板，并具备基于角色的权限控制。

3.2 主要内容
（1）梳理食堂/餐饮企业日常食品安全管理流程；
（2）完成需求、架构、数据库与页面设计；
（3）开发核心功能并进行测试；
（4）撰写毕业论文与答辩材料。

3.3 关键问题
（1）问题食品批次全生命周期管理建模；
（2）对接触该批次的环节/人员/档口进行排查；
（3）抽检结果与处置结论关联；
（4）通过可视化提升管理者风险感知能力。

四、技术方案与系统架构

4.1 技术选型
Spring Boot、Spring Security、MyBatis、MySQL、Thymeleaf、AdminLTE、jQuery、ECharts、PageHelper、Druid、Maven。

4.2 架构设计
经典三层架构与模块化分包（controller/service/dao/entity/config）。前端以管理后台形式提供业务页面，关键列表支持分页与条件查询。

4.3 安全与合规说明
系统管理业务台账，演示使用虚拟商户与虚拟批次；登录鉴权防止越权；重要处置操作保留操作者信息。

五、系统功能模块设计

5.1 食品安全风险看板
今日晨检异常人数、高风险批次数量、待处置隐患、抽检不合格数、物资库存预警、近7日风险趋势。

5.2 用户权限管理
管理员、食品安全员、档口负责人、普通员工等角色。

5.3 从业人员健康晨检
体温、有无腹泻呕吐等高风险症状、手部破损情况等；异常人员禁止上岗（状态标记）。

5.4 高风险食品批次管理
对疑似变质、投诉涉及、抽检异常的食品批次建档，记录供应商、生产日期、使用档口、风险等级、处置状态。

5.5 关联排查对象管理
排查使用过该批次的档口、菜品、责任人；建立观察/停用清单。

5.6 处置闭环管理
合格解除、退货销毁、停售整改等结局登记。

5.7 抽检与快检记录
农药残留、致病菌、添加剂等抽检结果登记（字段可简化），关联批次。

5.8 消杀与防护物资管理
消毒液、手套、体温枪、快检试剂等库存台账。

5.9 专题页
人员健康专题、抽检合格率专题、物资保障专题。

六、数据库设计

用户与角色、员工、晨检记录、食品批次、风险批次、关联排查、抽检记录、处置记录、物资与出入库、供应商字典。批次号建议唯一索引，便于追溯查询。

七、可行性分析

Spring Boot 技术成熟；经济上开源免费；操作上符合食堂管理员习惯；使用模拟数据，不涉及真实商户隐私。

八、进度安排

第1–2周：开题与文献；
第3–4周：流程调研与数据库设计；
第5–8周：权限、晨检、批次档案；
第9–11周：排查、抽检、处置、物资、看板；
第12–13周：联调测试；
第14–16周：论文与答辩。

九、预期成果

可运行系统、SQL 脚本、毕业论文、测试用例、演示账号与答辩 PPT。

十、创新点与特色

（1）以食品批次风险管控为核心建模；
（2）从业人员健康与食品追溯双主线；
（3）风险发现到销毁/解除的闭环；
（4）可视化支撑食堂自查与迎检。

十一、难点与对策

食品标准复杂：以通用抽检结果字段表达；
全链条溯源过重：先做到批次级与档口级；
明厨亮灶硬件视频：不纳入必做范围。
