本科毕业设计（论文）开题报告

题目：基于 Spring Boot 的医院感染防控管理系统的设计与实现

一、选题背景与意义

1.1 选题背景
医院感染防控是医疗质量与患者安全的重要组成部分。住院患者基础疾病多、侵入性操作频繁，叠加人员密集与设备共用，使医院成为感染风险高发场所。感控工作通常包括病例监测、接触者追踪、环境与手卫生核查、病原检测登记、隔离措施落实、转归复盘以及防护物资保障等环节。

传统依赖手工填报和纸质台账的方式，容易出现报告延迟、漏报错报、追踪困难、预警不足等问题。智慧医院建设要求将感控从“事后统计”推进到“过程管理与闭环处置”。因此，设计一套结构清晰、流程闭环、权限分明的医院感染防控管理系统，具有现实针对性与应用价值。

1.2 选题意义
（1）提升医院感染事件发现、报告、处置与复盘能力；
（2）以信息化手段固化感控流程，减少人为遗漏；
（3）综合运用 Spring Boot 技术栈完成完整工程训练；
（4）成果可迁移到基层医疗机构或教学演示场景。

二、国内外研究现状

2.1 国外研究现状
发达国家较早开展医院感染监测信息化，部分系统可从检验、手术、ICU 等系统抽取风险信号，实现感染预警与质量持续改进，并强调数据标准与隐私合规。

2.2 国内研究现状
国内三级医院普遍建设了院感模块或独立感控系统，但不同医院水平差异较大。中小医疗机构常面临预算有限、实施复杂、人员不足等问题，需要轻量级、易部署、主流程完整的管理系统。

2.3 本课题切入点
面向中小医院与教学演示场景，构建可运行的院感防控管理平台，突出暴露追踪、状态闭环、日常核查与综合看板。

三、研究目标与主要内容

3.1 研究目标
基于 Spring Boot 设计并实现医院感染防控管理系统，支持感控人员、科室管理员、普通医护等角色协作，完成高风险患者建档、接触者管理、转归跟踪、日常核查打卡、检测记录、防护物资管理与统计看板。

3.2 主要内容
需求分析、架构设计、数据库设计、核心模块开发、权限控制、测试验证、论文撰写。

3.3 拟解决的关键问题
（1）院感病例与接触者关联建模；
（2）患者状态流转可追溯；
（3）科室维度统计与基础预警；
（4）敏感医疗数据的权限隔离。

四、技术方案与系统架构

4.1 技术选型
后端：Spring Boot、Spring MVC、Spring Security
持久层：MyBatis、PageHelper
数据库：MySQL
连接池：Druid
前端：Thymeleaf、AdminLTE、jQuery、ECharts
构建：Maven；语言：Java 8/11

4.2 系统架构
采用 B/S 分层架构：表现层、业务层、持久层、数据层。业务按感控监测、病例管理、暴露追踪、物资保障、系统管理划分模块。

4.3 安全方案
Spring Security 登录认证与角色授权；越权拦截；会话管理；演示环境使用脱敏模拟数据。

五、系统功能模块设计

5.1 感控综合态势看板
新发院感数、在管病例、接触者数量、今日核查完成率、阳性检测数、物资低库存预警；趋势图与科室分布图。

5.2 系统用户与权限管理
用户维护、角色分配（感控管理员/科室用户/普通用户）、登录注销。

5.3 日常感控核查打卡
病区按日填报手卫生抽查、环境消毒、发热筛查等情况；异常记录检索。

5.4 院感病例/高风险患者管理
患者基本信息、科室床号、感染部位、病原体、隔离措施、病情变化记录。

5.5 暴露接触者管理
登记医护暴露、同病房患者等；观察期限管理；可转为病例或解除观察。

5.6 转归管理
治愈、死亡、转科转院等结局登记，形成闭环。

5.7 病原检测记录
送检项目、结果、报告时间，与患者关联并支持历史查询。

5.8 防护物资管理
口罩、防护服、手套、消毒液等入库出库与库存预警。

5.9 专题统计页
按科室、病原体、时间维度统计。

六、数据库设计思路

核心表：用户与权限、科室字典、患者基础信息、院感病例、接触者、核查打卡、检测记录、转归记录、物资与出入库。强调病例—接触者—检测—转归的关联完整性。

七、可行性分析

技术成熟、成本可控、界面符合医院后台使用习惯；以教学演示为目标，不接入真实 HIS，避免合规风险。

八、研究方法与进度安排

文献研究、需求调研、原型迭代、黑盒测试。

第1–2周：调研与开题；
第3–4周：需求与数据库设计；
第5–9周：Spring Boot 框架搭建与核心业务开发；
第10–12周：看板、物资、联调优化；
第13–14周：测试；
第15–16周：论文与答辩准备。

九、预期成果

可运行系统、数据库脚本、毕业论文、测试记录、答辩材料。

十、创新点与特色

（1）面向院感长期管理需求设计业务闭环；
（2）日常核查与事件处置一体化；
（3）暴露追踪流程可完整演示；
（4）角色权限适配感控与临床科室协同。

十一、可能存在的问题与对策

业务专业性强：以标准管理流程抽象，不深入临床医学争议；
功能范围易膨胀：优先保证主链路；
真实系统集成复杂：以独立系统+模拟数据完成目标。
